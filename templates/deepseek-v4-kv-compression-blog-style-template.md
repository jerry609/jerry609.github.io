# 技术机制长文模板：从误解到预算化形式化

基准文章：[DeepSeek-V4 的 KV 压缩：从删 token 误解到预算检索](https://jerry609.github.io/blog/deepseek-v4-kv-compression-budgeted-retrieval/)

适用对象：AI 系统、Transformer 机制、长上下文、推理系统、训练策略、工程源码等需要同时解释直觉、数学形式和实现路径的技术博客。

## 1. 文章定位

这类文章不是新闻解读，也不是只讲直觉的科普。它的写法是：

1. 先抓住一个常见但过度简化的判断。
2. 指出这个判断遗漏了哪些机制层级。
3. 给出更合适的建模方式。
4. 从基础公式开始，把机制逐步形式化。
5. 再落到训练、评估和源码路径，避免停留在概念层。

文章的核心姿态是：把一个容易被口号化的技术点，还原成可以检查的表示、约束、目标函数和实现路径。

## 2. 标题与摘要模板

标题建议使用：

```text
[系统/论文/机制名] 的 [核心机制]：从 [常见误解] 到 [更准确的建模方式]
```

示例结构：

```text
[模型名] 的 [机制]：从 [错误理解] 到 [预算/优化/检索/表示学习]
```

description 建议写成一句完整判断：

```text
从 [组件 A]、[组件 B] 和 [组件 C] 的分工出发，将 [现象/能力] 理解为 [数学或系统问题]。
```

## 3. 开头写法

开头采用“误解 -> 遗漏 -> 重新建模 -> 一句话概括”的顺序。

可套用骨架：

```markdown
[系统/模型] 的 [能力/现象] 常被简化为如下判断：

> [一句常见误解]

该判断省略了 [表示形态]、[预算约束] 和 [模块分工] 之间的关系。

更合适的建模方式是：[把现象写成一个计算问题/优化问题/检索问题]。

[系统/机制] 可以概括为：

> [三段式短句：局部怎么处理，远程怎么处理，相关项怎么处理]

需要注意的是，[被误解对象] 并不等于 [绝对结论]。它只是在 [当前层/当前 query/当前 head/当前分支/当前预算] 下 [未被读取/贡献较小/被压缩表示]。
```

开头语气要克制，不使用“彻底颠覆”“本质上就是”一类夸张判断。优先使用“更合适的建模方式是”“可以概括为”“需要注意的是”。

## 4. 全文结构模板

推荐章节顺序：

```markdown
## 符号表

## 1. [对象] 与表示形态

## 2. [近似/检索/压缩] 的成立条件

## 3. [未被选择/未被展开/被压缩对象] 去了哪里

## 4. [核心机制] 旁边还站着哪些配套机制

## 5. 训练阶段的适配机制

## 6. 形式化为优化问题

### A. [信息瓶颈 / rate-distortion / 表示压缩]

### B. [learned retrieval / routing / selection]

### C. [end-to-end loss minimization]

## 7. 选择质量评估

## 8. [系统名] 押了哪些经验判断

## 9. 官方源码路径

### 9.1 [配置如何决定路径]

### 9.2 [核心模块一做了什么]

### 9.3 [运行时何时读写状态]

### 9.4 [选择器/检索器如何工作]

### 9.5 [多个分支如何合并]

### 9.6 [kernel/底层实现实际读了什么]

### 9.7 [cache/state 如何分区]

### 9.8 [因果性/边界条件如何保证]

### 9.9 源码路径的最小伪代码

### 9.10 源码给出的若干结论

## 10. 总结
```

章节命名要直接，不追求文学化。每节标题回答一个具体问题，例如“成立条件”“去了哪里”“源码路径”，让读者能按问题索引。

## 5. 符号表规则

只要文章后面有多处公式，就先放符号表。符号表的作用不是装饰，而是让后续推导能连续阅读。

符号表推荐列：

```markdown
| 符号 | 含义 |
| --- | --- |
| $t$ | 当前生成位置，也就是当前 query 的位置 |
| $i,j,b$ | token、压缩块或 memory entry 的下标 |
| $\mathcal{P}_t$ | 位置 $t$ 可见的历史集合 |
| $q_t,k_i,v_i$ | query、key、value 向量 |
| $\alpha_{t,i}$ | softmax 后的 attention 权重 |
| $B_b$ | 第 $b$ 个压缩块包含的 token 下标集合 |
| $z_b$ | 第 $b$ 个 compressed memory entry |
| $S_t$ | 位置 $t$ 被选中的候选集合 |
| $M,C$ | memory 和 FLOPs 预算 |
| $\theta,\phi,\psi$ | 主模型、压缩器、检索器的参数 |
```

符号表写法：

1. 每个符号都要绑定一句自然语言解释。
2. 下标尽量稳定，不要在不同章节反复换记号。
3. 英文术语可以保留，例如 query、memory entry、top-k、KV cache，但第一次出现时用中文解释它在本文里的角色。
4. 与源码对应的符号要在后文明确映射到变量名或模块名。

## 6. 公式展开程度

公式展开采用“从完整形式到近似形式，再到误差或目标函数”的节奏。目标是让读者能跟着每一步，而不是只看到最终结论。

### 6.1 基础定义要完整写出

先写集合，再写打分，再写 softmax，再写输出：

```markdown
当前位置是 $t$，可见历史写成：

$$
\mathcal{P}_t=\{i:0\le i<t\}
$$

每个历史位置和当前 query 的打分是：

$$
e_{t,i}=q_t^\top k_i,\qquad i\in\mathcal{P}_t
$$

softmax 权重是：

$$
\alpha_{t,i}
=
\frac{\exp(e_{t,i})}
{\sum_{\ell\in\mathcal{P}_t}\exp(e_{t,\ell})}
$$

输出是：

$$
o_t=\sum_{i\in\mathcal{P}_t}\alpha_{t,i}v_i
$$
```

不要把四步压成一句“attention 就是 softmax(QK)V”。这篇风格强调可检查的变量流。

### 6.2 关键推导要逐行展开

当文章需要证明某个近似为什么成立时，使用 `aligned` 展开 3 到 5 行。

推荐模式：

```markdown
先将 full output 拆成选中部分和未选中部分：

$$
o_t
=
\sum_{i\in S}\alpha_{t,i}v_i
+
\sum_{i\in\mathcal{P}_t\setminus S}\alpha_{t,i}v_i
$$

定义被选中部分的质量 $A_t(S)$ 和未选中部分的残余质量 $R_t(S)$：

$$
A_t(S)=\sum_{i\in S}\alpha_{t,i},
\qquad
R_t(S)=1-A_t(S)
$$

于是误差可以写成：

$$
\begin{aligned}
\|o_t-\hat{o}_t\|
&= \|(1-R_t(S))o_S+R_t(S)o_{\bar S}-o_S\|\\
&= \|R_t(S)o_{\bar S}-R_t(S)o_S\|\\
&= R_t(S)\|o_{\bar S}-o_S\|.
\end{aligned}
$$
```

展开标准：

1. 第一次出现的集合、权重、损失都写定义。
2. 核心等式的变形保留中间步骤。
3. 最终不只给公式，还要用一句话解释它的判据含义。
4. 标准不等式可以写名字，例如“三角不等式”，但仍保留代入后的形式。
5. 如果有两个误差源，拆成两个命名项，例如 selection error 和 compression error。

### 6.3 公式密度

推荐密度：

1. 每个小概念 1 个 display formula。
2. 每个关键推导 3 到 7 个 display formula。
3. 每个 display formula 前要有一句“这一步在定义什么”。
4. 每个公式组后要有 1 到 3 句解释，说明它改变了什么问题。
5. 同一段不要连续堆超过 4 个公式组；中间插入自然语言解释。

不推荐：

1. 直接给最终 boxed 结论而不铺垫。
2. 一页内同时切换过多记号。
3. 用“显然”“易得”跳过本文的主推导。
4. 把源码变量、论文符号和自定义符号混在一起而不映射。

## 7. 机制解释句式

这篇文章常用的句式是“局部限定 + 机制拆分 + 谨慎结论”。

可复用句式：

```text
该判断省略了 [A]、[B] 和 [C] 之间的分工。

更合适的建模方式是：[现象] 可视为 [带约束的问题]。

[对象] 不是没有贡献，而是没有在 [当前条件] 下以 [原始形态] 被读取。

若表述为 “[错误说法]”，会混淆 [概念 A] 与 [概念 B]。

更贴近实际机制的描述是：[准确说法]。

于是问题变成两个小问题。

该结论不能外推为永久判决。

该方案没有将全部压力放在 [单一机制] 上。

质量来自训练阶段对 [读取方式/表示格式/预算约束] 的适配。

需要限定分析范围：[源码/报告/实验] 能说明 [A]，但不能单独证明 [B]。

该结果说明 [观察]，但提升不能全部归因于 [单一机制]。
```

措辞偏好：

1. 使用“可视为”“可以写成”“可拆成”“更贴近实际机制”。
2. 对经验判断使用“通常”“往往”“可能”“在该数据分布下”。
3. 对局部性使用“当前层、当前 query、当前 head/分支”这样的限定语。
4. 对机制关系使用“分工”“表示形态”“读取预算”“候选集合”“边界条件”。
5. 对工程实现使用“源码路径可简化为”“代码路径对应公式中的”。

避免措辞：

1. 不说“完全等价”，除非有严格证明。
2. 不说“无损压缩”，除非机制确实保证无损。
3. 不说“所有 token 都没用”这类绝对判断。
4. 不把 benchmark 提升单独归因给一个组件。
5. 不使用营销式形容词，例如“革命性”“吊打”“史诗级”。

## 8. 解释源码的写法

源码段落遵循“先限定范围，再给代码骨架，再映射公式，再给结论”的顺序。

可套用骨架：

````markdown
前文提供了机制层面的解释。[官方仓库/实现文件] 能够将该机制对应到具体读写路径。

需要限定分析范围：[这个目录/文件] 适合观察 [推理/训练/缓存/调度]，但不代表 [完整服务端/完整训练系统]。

核心路径可简化为：

```python
[最小伪代码]
```

对应公式可以分几步。

第一步，[源码操作]：

$$
[公式]
$$

第二步，[源码操作]：

$$
[公式]
$$

因此，源码里的 [模块] 可视为 [机制角色]。
````

源码伪代码要短，只保留主干。量化、并行、kernel 优化、异常分支等细节可以在伪代码后说明“这里省略了什么”。

## 9. 评估章节写法

评估不是只列指标，而是说明每个指标检查什么前提。

推荐结构：

```markdown
### 1. Captured [Mass/Recall/Signal]

该指标检查：[核心假设] 是否成立。

$$
[指标定义]
$$

[指标] 越高，说明 [选择器/压缩器/路由器] 在该数据分布下越接近 [oracle/完整机制]。

### 2. Oracle vs learned

先定义 oracle：

$$
[oracle selection]
$$

再定义 model selection：

$$
[model selection]
$$

比较方式是：

$$
[Recall@k / overlap / rank correlation]
$$

需要注意：[这个指标能说明什么]，但不能直接推出 [更强结论]。
```

评估章节要主动写出局限性，尤其是：

1. 指标验证的是哪个子路径。
2. 它不能证明哪个整体结论。
3. benchmark 提升还有哪些混杂因素。
4. 哪些 worst-case 会击穿前提。

## 10. 经验判断章节写法

用编号列出系统设计背后的假设，每条都写成可反驳的经验判断。

模板：

```markdown
该设计背后的经验判断可拆成几条：

1. [远程/全局/稀疏] 依赖通常 [性质]。
2. [局部/短程] 依赖仍然重要，因此需要 [机制]。
3. [远处/低频/背景] 信息有相当一部分可以 [摘要化/聚合/压缩]。
4. 训练能让模型适应 [新的表示格式/读取约束]。
5. 真实任务分布通常避开 [最坏情况]；当任务要求 [worst-case 行为] 时，近似误差会增大。
```

这部分要像“设计假设列表”，不要写成宣传结论。

## 11. 总结写法

总结先用一段自然语言归纳，再给流程公式，最后给 boxed 结论。

可套用骨架：

```markdown
[机制] 可以总结为：

> 在给定 [位置/查询/预算] 上，[大多数对象] 不需要以 [原始形态] 被逐个读取；它们可以由 [摘要/压缩表示] 覆盖，或者对 [当前目标] 的边际贡献不足以占用预算。真正要优化的是在 [资源约束] 下，让 [训练目标/输出分布] 尽量接近 [完整模型/理想机制]。

先有 [输入表示]：

$$
[H_{1:T}]
$$

再做 [压缩/路由/选择]：

$$
[z_b=\Phi_\phi(H_b)]
$$

当前 query 做预算内检索：

$$
[S_t=\operatorname{TopK}_k\{g_\psi(q_t,z_b)\}_b]
$$

最后用 [训练目标] 约束整套读取方式：

$$
[\mathcal{L}=...]
$$

合在一起就是：

$$
\boxed{
[机制]
=
[组件 A]
+
[组件 B]
+
[组件 C]
}
$$
```

boxed 结论要短，最好是三个名词短语相加。它是收束，不是新概念引入。

## 12. 写作检查清单

发布前检查：

1. 开头是否明确指出一个常见误解。
2. 是否在前三段给出更准确的建模方式。
3. 是否有符号表，并且后文符号没有漂移。
4. 基础公式是否从集合、打分、权重、输出逐步展开。
5. 关键推导是否保留中间等式，而不是只给结论。
6. 每个公式组后是否有自然语言解释。
7. 是否区分了 selection error、compression error、training adaptation、implementation path。
8. 是否写清楚“当前层/当前 query/当前 head/当前预算”这类局部限定。
9. 是否避免把局部指标解释成整体等价。
10. 源码段落是否先限定范围，再给伪代码和公式映射。
11. benchmark 或报告数字是否只支持其能支持的结论。
12. 总结是否回到“资源预算下的目标函数优化”，而不是停在一句口号。
